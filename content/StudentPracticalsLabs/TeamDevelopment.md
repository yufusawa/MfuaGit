## Самостоятельный проект "Командная работа над git-проектом"

> ### Совместная разработка на проекта на git — это процесс, когда несколько разработчиков работают над одним проектом, используя инструменты GitHub для координации и управления изменениями кода.

Преимущества совместной разработки:
* **Прозрачность** — вся история изменений доступна
* **Code Quality** — ревью кода улучшает качество
* **Knowledge Sharing** — коллективное обучение
* **Backup** — код хранится удаленно
* **Координация** — меньше казусов между разработчиками

> ### Всю работу выполнять в VSCode и Github!

Студентам:
- Создать команды по 3-4 человека, назначить среди них тимлида (командира команды). Название команды и список участников с указанием тимлида прислать в чат встречи;
- Вместе придумывают проект, например - "Сайт компании по разработки Телеграм-ботов" (ничего шокирующего);
- Тимлид создаёт публичный репозиторий на **Github** с `README.md` и приглашает выбранных соразработчиков для совместного кодинга в Git-сервис в роли **Collaborator** (Collaborator);
- Сайт статический, **HTML+CSS+JS** на 5-6 страниц с картинками для **Github Page**;
- Договориться между собой в **Телеграме** или в **Team**s, кто какую работу по созданию сайта будет выполнять;
- Создать дополнительную ветку для разработки сайта, например **develop**;
- Создать сайт;
- Красиво оформить `README.md` проекта, с картинками, списками, ссылкой на сайт проекта и диаграмой на **Mermaid**;
- Включить технологию **Github Page** для получения ссылки на сайт;
- Прислать мне 2 ссылки на готовый сайт (на репозиторий с README.md и сайт).

### Примерные ветки проекта
* main/master    - стабильная версия
* develop        - разработка
* feature/*      - новые функции (не обязательно)
* hotfix/*       - срочные исправления (не обязательно)

```shell

```

### Полезные команды для работы над проектом

Постоянная синхронизация перед каждым изменением в проекте:

```shell
git pull
```

или (более мягкий вариант)
```shell
git fetch
```

Просмотр изменений:
```shell
git log --oneline --graph --all    # визуализация истории
git diff                           # текущие изменения
git diff --staged                  # изменения в индексе
```

### Работа с удаленным репозиторием
```shell
git remote -v                      # список удаленных репозиториев
git branch -r                      # удаленные ветки
```

### Решение возможных проблем

- Чаще выполняйте синхронизацию репозиториев командой `git pull`
- Договаривайтесь между участниками перед выполнением `git push`
- Работайте в разный ветках проекта, чтобы потом объединить их с главной веткой
- Если появилась ошибка слияния, то выбирайте `rebase false` и `git pull`
- Используйте нейросеть для помощи
