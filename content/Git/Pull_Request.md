# Pull Request (PR): полное руководство

Pull Request (PR) — это центральный элемент совместной разработки на GitHub (и аналогичных платформах: GitLab, Bitbucket). Это не просто техническая операция слияния веток, а полноценный процесс **обсуждения, рецензирования и согласования изменений** перед их включением в основную кодовую базу.

## 📌 Что такое Pull Request?

**Pull Request** (буквально «запрос на подтягивание») — это предложение внести изменения из одной ветки (обычно тематической, feature-ветки) в другую (например, `main` или `develop`). Когда вы создаёте PR, вы просите других участников проекта просмотреть ваш код, обсудить его и, если всё хорошо, влить (merge) в целевую ветку.

PR объединяет в себе:
- **Код** (сами изменения).
- **Обсуждение** (комментарии, вопросы, предложения).
- **Историю** (все коммиты и обновления).
- **Интеграцию** (автоматические проверки, CI/CD).

## 🔍 Зачем нужен Pull Request?

1. **Контроль качества кода (Code Review)**
   Другие разработчики могут проверить код на ошибки, соответствие стандартам, архитектурные проблемы. Это повышает качество и снижает риск багов.

2. **Обсуждение и обмен знаниями**
   В PR можно задавать вопросы, предлагать альтернативные решения, обсуждать детали реализации. Это помогает команде лучше понимать код и делиться опытом.

3. **Автоматические проверки (CI/CD)**
   При создании PR обычно запускаются тесты, линтеры, сборка проекта. Если проверки не проходят, PR нельзя смержить (или это требует отдельного решения).

4. **Документирование изменений**
   Описание PR и комментарии остаются в истории — это документация того, **почему** были сделаны те или иные изменения.

5. **Безопасность и контроль доступа**
   Не все могут пушить напрямую в основную ветку. PR — единственный способ внести изменения, что защищает проект от случайных поломок.

## 🛠 Как создать Pull Request (пошагово)

### Предварительные шаги (локально)

1. **Обновите локальную основную ветку**
   ```bash
   git checkout main
   git pull origin main
   ```

2. **Создайте новую ветку для своей задачи**
   ```bash
   git checkout -b feature/awesome-feature
   ```

3. **Внесите изменения и сделайте коммиты**
   ```bash
   git add .
   git commit -m "feat: add awesome feature"
   ```

4. **Отправьте ветку в удалённый репозиторий**
   ```bash
   git push origin feature/awesome-feature
   ```

### Создание PR на GitHub

1. После пуша GitHub часто предлагает кнопку **"Compare & pull request"**. Если нет — перейдите в раздел **"Pull requests"** и нажмите **"New pull request"**.

2. Выберите **base** (целевую ветку, обычно `main`) и **compare** (вашу ветку).

3. Заполните форму:
   - **Заголовок** — кратко описывает суть изменений.
   - **Описание** — подробно: что сделано, зачем, как тестировалось. Можно использовать Markdown, ссылаться на issue (#123).
   - **Reviewers** — укажите, кто должен проверить код.
   - **Assignees** — назначьте ответственного (обычно себя).
   - **Labels** — добавьте метки (например, `enhancement`, `bug`).
   - **Projects** — привяжите к канбан-доске, если используется.

4. Нажмите **"Create pull request"**.

### Черновик (Draft PR)

Если вы ещё не закончили работу, но хотите показать код или обсудить подход, создайте **Draft PR** (кнопка "Create draft pull request"). Такой PR нельзя смержить, пока он не переведён в обычный.

## 🔄 Жизненный цикл Pull Request

### 1. Code Review

Ревьюверы оставляют комментарии:
- **Общие комментарии** (к PR в целом).
- **Комментарии к строкам кода** — указывают на конкретные места.

Типы комментариев:
- **Comment** — просто замечание.
- **Request changes** — требует исправлений (блокирует мерж).
- **Approve** — одобрение, можно мержить.

### 2. Обсуждение и доработка

- Автор отвечает на комментарии, обсуждает.
- При необходимости вносит правки в код:
  ```bash
  # вносим изменения
  git add .
  git commit -m "fix: review comments"
  git push origin feature/awesome-feature
  ```
  PR автоматически обновляется новыми коммитами.

### 3. Повторное ревью

После внесения правок ревьюверы смотрят изменения и либо одобряют, либо просят ещё доработок.

### 4. Проверки CI

Если в проекте настроены автоматические проверки (GitHub Actions, Travis, Jenkins), они запускаются при каждом обновлении PR. Статус (✅ успех / ❌ ошибка) отображается прямо в PR. Мерж возможен только при успешных проверках (если это настроено).

### 5. Мерж (слияние)

Когда PR одобрен и все проверки пройдены, его можно смержить. GitHub предлагает три стратегии:

- **Create a merge commit** — создаёт merge-коммит, сохраняя историю ветки.
- **Squash and merge** — объединяет все коммиты ветки в один коммит. Удобно для чистоты истории.
- **Rebase and merge** — перебазирует коммиты поверх целевой ветки (без merge-коммита). Требует отсутствия конфликтов.

Выбор зависит от политики команды. После мержа можно удалить ветку (кнопка **"Delete branch"**).

### 6. Закрытие PR без мержа

PR можно закрыть без слияния (например, если изменения больше не нужны). Просто нажмите **"Close pull request"**.

## 🎯 Best Practices для работы с PR

### Для автора

- **Делайте небольшие PR** — один PR = одна задача. Легче ревьюить и меньше конфликтов.
- **Пишите хорошее описание** — объясните **почему** сделаны изменения, а не только **что**.
- **Ссылайтесь на issues** — например, `Closes #123`. При мерже issue закроется автоматически.
- **Обновляйте ветку** — регулярно подтягивайте изменения из целевой ветки, чтобы избежать конфликтов:
  ```bash
  git fetch origin
  git rebase origin/main
  git push --force-with-lease  # если rebase, нужен force push
  ```
- **Отвечайте на комментарии** — будьте вежливы, благодарите за замечания.

### Для ревьювера

- **Будьте конструктивны** — критикуйте код, а не автора.
- **Предлагайте альтернативы** — не просто "это плохо", а "может, лучше сделать так?".
- **Не затягивайте** — старайтесь ревьюить быстро, чтобы не блокировать коллегу.

### Общие

- **Используйте шаблоны PR** — в репозитории можно создать файл `.github/PULL_REQUEST_TEMPLATE.md`, который автоматически подставит структуру описания.
- **Включайте автоматические проверки** — тесты, линтеры должны запускаться на PR.
- **Обсуждайте архитектурные вопросы до кода** — иногда лучше сначала обсудить подход в issue, а потом писать код.

## 📦 Примеры команд Git, полезных при работе с PR

```bash
# Создать ветку
git checkout -b feature/new

# Отправить ветку в remote
git push origin feature/new

# Получить обновления из main и перебазироваться
git fetch origin
git rebase origin/main

# Если rebase потребовал force push (после перебазирования)
git push --force-with-lease origin feature/new

# Посмотреть, что изменилось в PR (после fetch)
git log main..origin/feature/new
git diff main...origin/feature/new
```

## 🌐 Связь с Issues

Вы можете связать PR с issue несколькими способами:
- В описании PR написать `Fixes #123` или `Closes #123`.
- На GitHub в разделе "Development" можно связать issue вручную.

При мерже PR с ключевым словом `Closes` issue автоматически закроется.

## 🧪 Draft PR (черновик)

Используйте Draft, когда:
- Вы хотите показать код на ранней стадии.
- Нужна помощь или совет до завершения.
- Код ещё не готов к мержу.

Перевести из Draft в обычный PR можно через интерфейс (кнопка "Ready for review").

## 🚫 Что не стоит делать

- Не создавайте гигантские PR на тысячи строк — их никто не будет ревьюить.
- Не игнорируйте комментарии ревьюверов.
- Не мержите PR с падающими тестами (если это не согласовано).
- Не используйте `git push --force` без `--force-with-lease` и без предупреждения.

## 📈 Заключение

Pull Request — это не просто технический инструмент, а основа культуры современной разработки. Он превращает процесс интеграции кода в открытое обсуждение, улучшает качество и сплачивает команду. Освоив PR, вы сможете эффективно работать над любыми проектами — от open-source до корпоративных.

**Главные принципы:**
✅ Маленькие, понятные изменения
✅ Конструктивное общение
✅ Автоматизация проверок
✅ Уважение к чужому времени
